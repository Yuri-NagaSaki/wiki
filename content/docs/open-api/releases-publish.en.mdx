---
title: Publish a release
description: POST /api/releases/publish — the authenticated write endpoint groups use to upload torrents; supports JSON (magnet) and multipart/form-data (torrent file).
---

`/api/releases/publish` is the only **write** endpoint a release group uses. Authenticate by passing the group's API key in the request header. Both `application/json` (magnet link) and `multipart/form-data` (torrent file upload) are supported.

<Callout type="warn">
  This is a **write endpoint**: rate limits and quotas are tracked per group, separate from the IP-based RSS rate window. Rotate the key immediately if it leaks.
</Callout>

## URL

```http title="HTTP"
POST https://site.anibt.net/api/releases/publish
Authorization: Bearer <YOUR_API_KEY>
```

## Two request shapes

<Tabs items={['multipart/form-data (recommended)', 'application/json']}>
  <Tab value="multipart/form-data (recommended)">
    Upload the torrent and the backend **automatically extracts** the magnet, infoHash, and file size from it.

    ```bash title="Upload a torrent"
    curl -X POST 'https://site.anibt.net/api/releases/publish' \
      -H 'Authorization: Bearer <YOUR_API_KEY>' \
      -F 'bgmId=400602' \
      -F 'title=[Group] Frieren - 01 [1080p].mkv' \
      -F 'episodeKey=01' \
      -F 'resolution=1080p' \
      -F 'language=CHS' -F 'language=JP' \
      -F 'subtitle=EXTERNAL' \
      -F 'format=MKV' \
      -F 'torrent=@/path/to/release.torrent'
    ```

    Repeat `language` to send multiple values (CHS + JP).
  </Tab>
  <Tab value="application/json">
    Send the magnet link directly (you still need a torrent on disk — only well-formed magnets are accepted).

    ```bash title="JSON body"
    curl -X POST 'https://site.anibt.net/api/releases/publish' \
      -H 'Authorization: Bearer <YOUR_API_KEY>' \
      -H 'Content-Type: application/json' \
      --data @- <<'JSON'
    {
      "bgmId": 400602,
      "title": "[Group] Frieren - 01 [1080p].mkv",
      "magnetBase64": "magnet:?xt=urn:btih:...",
      "episodeKey": "01",
      "resolution": "1080p",
      "language": ["CHS", "JP"],
      "subtitle": "EXTERNAL",
      "format": "MKV",
      "fileSize": 734003200,
      "notes": "**Fonts embedded**, dropped frames fixed."
    }
    JSON
    ```

    `magnetBase64` accepts both the raw magnet URL and a Base64-encoded magnet.
  </Tab>
</Tabs>

  `torrent` is **required** in multipart requests. `magnetBase64` is optional — when omitted, the backend extracts it from the torrent.
</Callout>

## Preview mode

To verify how a release will render (title parsing, Markdown notes, magnet correctness) before going public, set `preview=true`:

```bash title="Preview only — not published"
curl -X POST 'https://site.anibt.net/api/releases/publish' \
  -H 'Authorization: Bearer <YOUR_API_KEY>' \
  -F 'bgmId=400602' \
  -F 'title=[Group] Frieren - 01 [1080p].mkv' \
  -F 'preview=true' \
  -F 'torrent=@/path/to/release.torrent'
```

The returned preview URL **expires in 10 minutes** and **does not** show up in RSS / search / public listings. Useful as the last manual gate in an automated pipeline.

## Authentication & key management

<Steps>
  <Step>
    ### Generate an API key in the group dashboard
    Each group can have multiple keys; consider one for CI and one for ad-hoc scripts.
  </Step>
  <Step>
    ### Send it on every request
    ```http
    Authorization: Bearer <YOUR_API_KEY>
    ```
  </Step>
  <Step>
    ### Rotate immediately on leak
    The previous key is revoked instantly; the new key is shown only once at generation, so save it carefully.
  </Step>
</Steps>

<Callout type="warn">
  Never embed keys in client-side code, commit them to a repo, or paste them in issues. Inject through CI secrets only.
</Callout>

## Common errors

| HTTP | Cause                                                  | Fix                                                            |
| ---- | ------------------------------------------------------ | -------------------------------------------------------------- |
| 400  | Missing `title` / `bgmId`, or torrent could not be parsed | Check required fields; ensure a valid .torrent file              |
| 401  | `Authorization` missing or malformed                    | The `Bearer ` prefix (with a space) is mandatory                |
| 403  | API key invalid / revoked / does not belong to the group | Generate a new key                                              |
| 409  | A release with the same infoHash already exists         | To replace, re-publish with `version=v2`                        |
| 413  | Torrent / form too large                                | Trim the tracker list inside the torrent and retry              |
| 429  | Write quota exhausted                                   | Back off using `Retry-After` from the response                  |
