## 各端点的上限一览

| 端点                    | `limit` 默认 | `limit` 上限 | 后端候选扫描上限 | 速率限制（默认）   |
| ----------------------- | ------------ | ------------ | ---------------- | ------------------ |
| `/rss/anime.xml`        | **100**      | **1000**     | **4000** 条候选  | 120 次 / 60 秒 / IP |
| `/rss/group/<slug>.xml` | 100          | **200**      | —                | 120 次 / 60 秒 / IP |
| `/rss/magnets.xml`      | 100          | **200**      | —                | 120 次 / 60 秒 / IP |
| `/api/subtitle-groups`  | —            | 全量返回     | —                | 同上 60 秒窗口      |
| `/api/seasons/anime`    | —            | —            | —                | 30 次 / 60 秒 / IP  |
| `/api/anime/groups`     | —            | —            | —                | 30 次 / 60 秒 / IP  |

<Callout type="info">
  **`limit` 越界处理**：超过上限会被自动**截断到上限**，不会报错。例如 `?limit=5000` 给 `/rss/anime.xml` 实际返回 1000 条。
</Callout>

### `MAX_OVERSCAN`（仅 `/rss/anime.xml`）

当你叠加 Tag 过滤（resolution / language / subtitle / format / episode / episodeKey）时，后端先按索引拉候选，再在内存里筛 Tag。**候选池上限是 4000 条**，避免单次请求扫库失控。

绝大多数番剧 + 字幕组组合的实际数据量都远小于这个值；如果你过滤极其严格（例如某番剧某组只有 3 集 1080p 简中外挂），后端拉到 4000 条后就停止，再做 Tag 筛选并返回 `limit` 条。**实操影响**：极冷门组合可能"看起来少"——这是设计上的安全阀。

## 提高上限

<Tabs items={['客户端层面', '服务端层面（自部署）', '触发 429']}>
  <Tab value="客户端层面">
    1. **拉满 `limit`**：把 URL 的 `?limit=` 改成端点上限（1000 / 200）。一次请求拿更多，等于把 N 次轮询合成 1 次。
    2. **用 304 替代轮询**：每次请求带 `If-None-Match`。服务端确认数据没变就返回 304 空 body，**不计入很多客户端的"流量统计"**，但仍会消耗 1 次速率配额。
    3. **拉长间隔**：默认建议 ≥ 5 分钟。番剧发布频率本身按"集数"计算，每分钟扫一次没有意义。
    4. **合并并发**：不要对同一 URL 并发发多个请求，缓存层会让后续请求拿到一样的结果，但白白消耗速率额度。
  </Tab>
  <Tab value="服务端层面（自部署）">
    通过 Convex / 部署平台的**环境变量**调整（重启后生效）：

    ```bash title="env"
    # 把每 IP 60 秒 120 次提高到 60 秒 600 次
    RSS_ANIME_RATE_LIMIT_MAX=600
    RSS_GROUP_RATE_LIMIT_MAX=600
    RSS_MAGNETS_RATE_LIMIT_MAX=600

    # 或者改窗口为 10 秒（变成 120 次 / 10 秒）
    RSS_RATE_LIMIT_WINDOW_MS=10000

    # 关闭 CDN 边缘缓存（不推荐，仅测试用）
    RSS_CACHE_TTL_SECONDS=0
    ```

    <Callout type="warn">
      `/rss/anime.xml` 的 `MAX_LIMIT=1000` 和 `MAX_OVERSCAN=4000` 是源码常量（`convex/rssFeed.ts`），**不通过环境变量调整**。如果你的私有部署需要更高，直接改源码后 `npx convex deploy`。
    </Callout>

    提高之后需要观察：

    - **单次响应 XML 大小**：1000 条约 0.8–2 MB，注意带宽。
    - **Convex query 时长**：候选 4000 条 + 内存 Tag 过滤通常 < 200ms；继续上调 `MAX_OVERSCAN` 接近函数 1 秒上限时会有失败风险。
    - **客户端解析时长**：很多 RSS reader 默认只保留最近 N 条，请同步调整。
  </Tab>
  <Tab value="触发 429">
    - 响应包含 `Retry-After: <seconds>`，按这个退避，**不要立即重试**。
    - 短期内被打 429 通常是配置错（轮询太快 / 多个 reader 用同 URL）。把间隔放大到 5 分钟 + 启用 304，几乎不会再触发。
  </Tab>
</Tabs>

## 1000 用户场景下的真实负载

> **核心结论**：origin 真实承受的回源率与并发用户数**几乎无关**，只与 `s-maxage` 和查询参数多样性有关。

请求流转：

```txt
用户 ──► 浏览器 / RSS 客户端缓存 (max-age)
        ──► Cloudflare CDN (s-maxage + SWR)
            ──► origin (Convex / TanStack SSR) ──► 数据库
```

- CDN 内未过期 → **0 origin 请求**
- CDN 过期但仍在 SWR 窗口 → 用户拿旧版本（0 延迟），origin 收到 **1 个**异步刷新请求
- 完全过期 → origin 收到 1 个同步请求（~50–200ms 回源）
- 客户端带 `If-None-Match` → ETag 命中时 origin 仅返回 304（无 body，几乎零开销）

各端点在 **1000 用户、客户端默认 15 分钟轮询** 假设下的预估负载：

| 端点                                     | 客户端日请求量    | origin 真实回源（次/小时） | 备注                                          |
| ---------------------------------------- | ----------------- | -------------------------- | --------------------------------------------- |
| `/rss/anime.xml`（全站）                  | ~96k              | **~12**                    | 5min CDN 窗口，全部用户共享同一缓存            |
| `/rss/group/<slug>.xml`                  | 取决于 slug 数    | **~12 × slug 数**          | 50 个 slug ≈ 600 次/小时                      |
| `/rss/magnets.xml`（简单）                | —                 | **~30**                    | 全部用户用相同筛选时                          |
| `/rss/magnets.xml`（复杂、按 query 分缓存） | —                 | **接近 1× 唯一查询数 / 60s** | 每人查询条件不同时 CDN 命中率 ≈ 0，SWR 缓解延迟 |
| `/api/subtitle-groups`                   | ~96k              | **~12**                    | 数据极少变化，几乎全 304                       |
| `/api/seasons/anime`                     | ~48k              | **~6**                     | 季度数据周更，长缓存                           |
| `/api/anime/groups`（按 bgmId）           | —                 | **~12 × 热门番剧数**        | 冷门番剧首次回源较多                          |

**综合**：1000 用户全套订阅 + 前端使用，origin 一小时实际承受 **< 1000 次请求**，其中 90%+ 是 304。SWR 让"用户感知延迟"和"origin 回源压力"完全解耦——回源慢不会拖慢用户。

<Callout type="warn">
  **绝对不要做的事**：
  - 多线程并发拉同一 URL（白白消耗速率配额，CDN 会让结果完全相同）
  - 关闭 ETag / `If-None-Match`（每次都拿全量响应，浪费带宽）
  - 间隔 < 1 分钟（必触发 429，且对你没任何收益）
</Callout>

## 主流 RSS 客户端的实际行为

不同客户端默认的轮询频率、ETag/304 支持差异很大。**好消息**：本站对所有客户端都返回正确的 `ETag` + `Last-Modified` + `Cache-Control`，符合 HTTP 标准的客户端会自动走 304，**你不用做任何配置**。

| 客户端                              | 默认轮询           | `If-None-Match` | 尊重 `Cache-Control: max-age` | 备注                                            |
| ----------------------------------- | ------------------ | --------------- | ----------------------------- | ----------------------------------------------- |
| **NetNewsWire**（macOS/iOS）        | 1 小时             | ✅              | ✅                            | 表现最好，几乎全 304                             |
| **Inoreader**（云端）               | ~30 分钟           | ✅              | ✅                            | 自带服务端聚合，1000 用户共享同一上游请求         |
| **Feedly**（云端）                  | ~1 小时            | ✅              | ✅                            | 同上，对 origin 极友好                          |
| **FreshRSS**（自建）                | 用户配置（默认 1h）| ✅              | ✅                            | 推荐配置 ≥ 15min                                |
| **Tiny Tiny RSS**（自建）           | 用户配置           | ✅              | ⚠ 部分版本无视                 | 建议手动设 ≥ 15min                              |
| **Reeder**（macOS/iOS）             | 跟随 Inoreader/Feedly | —            | —                             | 透传第三方聚合服务                              |
| **qBittorrent RSS 自动下载**         | **默认 30 分钟**   | ❌（每次拉全量） | ❌                            | **对 origin 最不友好**，但配合 CDN 仍 OK         |
| **Transmission + Flexget**          | 用户配置           | ✅（Flexget 支持）| ✅                           | 推荐                                            |
| **自建脚本（Python `requests`）**    | 你写多少就多少     | 取决于实现      | 取决于实现                    | 见 [Python 客户端最小实现](/docs/open-api/caching#python-客户端最小实现) |

### 针对场景的预期

- **个人订阅 1–2 个番剧 RSS（NetNewsWire / Inoreader）**：每小时 1 个请求，全部 304，origin 几乎感知不到。
- **qBittorrent 用 RSS 规则自动下载**：每 30 分钟拉一次全量 XML（约 50–200 KB），但因为 CDN 命中，实际打到 origin 也只是 5 分钟一次。
- **公开聚合站（Inoreader / Feedly）**：1000 个订阅者后端共享同一份订阅，对 origin 等同 1 个用户。

### 调优建议

- 用 **Inoreader / Feedly / NetNewsWire iCloud** 这类支持云端聚合的客户端 → 你和其他用户共享缓存
- qBittorrent / Transmission 直接拉的话，把规则间隔调到 ≥ 15 分钟（默认 30 分钟也很合理）
- 自建脚本必须实现 `If-None-Match` 复用，否则等于浪费带宽

## 后端发布 → 用户看到的完整延迟拆解

以"字幕组发布一个新种子"到"订阅者的 RSS 客户端弹出通知"为例，逐段算一遍最坏情况：

```txt title="延迟拆解"
T=0       字幕组提交 release 到 AniBT 后台
          (Convex mutation 写入 releases / animeReleases / files 表)
T=0       数据库立即可读（Convex 强一致）
          /rss/anime.xml 此刻请求会拿到新数据，但 CDN 还缓存着旧版本

T≤300s    CDN 缓存（s-maxage=300）窗口内：
          客户端请求被 CDN 拦截，仍拿旧版本
          → 这一段是用户感知延迟的主要来源

T=300s    CDN 缓存进入 SWR 期：
          下一个用户请求触发 CDN 异步回源
          原 origin 拿到新数据 → CDN 缓存更新

T=300~600s 客户端的下次轮询周期（取决于客户端）：
          - 若客户端轮询间隔 = 5min → 最快下一次拿到新版本
          - 若间隔 = 15min → 平均再等 7.5 min
          - 若间隔 = 1h → 平均再等 30 min
```

### 典型场景的端到端延迟（中位数）

| 客户端配置                  | 后端发布 → 用户看到        |
| --------------------------- | -------------------------- |
| 自建脚本 5 min 轮询 + ETag  | **3–5 分钟**               |
| NetNewsWire 默认 1h 轮询    | 30–35 分钟（瓶颈在客户端） |
| Inoreader 云端聚合          | 15–30 分钟（瓶颈在 Inoreader 自己的轮询频率） |
| qBittorrent 默认 30min      | 15–30 分钟                 |

### 优化哪一段最有效

1. **缩短客户端轮询间隔（最有效）**：从 1h 改成 5min，立刻把延迟从 30min 砍到 5min。
2. **缩短 `s-maxage`（次要）**：从 300s 改成 60s，origin 压力涨 5 倍但用户延迟只少 4 分钟，性价比低。
3. **强制 origin 立即推送（不可行）**：HTTP RSS 是 pull 模型，没办法。

<Callout type="info">
  如果你需要**实时**（< 1 分钟）通知，RSS 不是合适方案，应该用 webhook / Telegram bot / 站内推送。RSS 的设计目标就是"**最终一致 + 极低成本**"，5–30 分钟的延迟是它的功能不是 bug。
</Callout>
