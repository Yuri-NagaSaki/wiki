AniBT 所有公开端点（RSS + `/api/*`）都返回三层缓存指令，并支持 `If-None-Match` / `If-Modified-Since` 条件请求。**配合标准 HTTP 客户端无需任何配置即可生效。**

<Callout type="info">
  按 RFC 7232，`If-None-Match`（ETag）优先于 `If-Modified-Since`。如果你同时发了两个头，服务端只看 ETag。建议永远用 ETag。
</Callout>

## 响应头

```txt title="所有端点统一返回"
Cache-Control:    public, max-age=<N>, s-maxage=<N>, stale-while-revalidate=<M>
CDN-Cache-Control: public, max-age=<N>, stale-while-revalidate=<M>
ETag:             W/"..."
Last-Modified:    <RFC 1123 date>
Vary:             Accept-Encoding
```

## 各端点 TTL 一览

| 端点                            | `max-age` (浏览器) | `s-maxage` (CDN) | `stale-while-revalidate` |
| ------------------------------- | ------------------ | ---------------- | ------------------------ |
| `/rss/anime.xml`                | 300s               | 300s             | 86400s                   |
| `/rss/group/<slug>.xml`         | 120s               | 300s             | 86400s                   |
| `/rss/magnets.xml`（简单）      | 120s               | 120s             | 600s                     |
| `/rss/magnets.xml`（复杂查询）  | 60s                | 60s              | 600s                     |
| `/api/subtitle-groups`          | 300s               | 300s             | 86400s                   |
| `/api/seasons/anime`            | 600s               | 600s             | 86400s                   |
| `/api/anime/groups`             | 300s               | 300s             | 3600s                    |

## ETag 计算公式

| 端点                                              | 公式                                                              |
| ------------------------------------------------- | ----------------------------------------------------------------- |
| `/rss/anime.xml` `/rss/group` `/rss/magnets.xml`  | `W/"r{itemCount}-{latestPublishedAtMs}-{canonicalUrlHash}"`       |
| `/api/subtitle-groups`                            | `W/"sg{count}-{maxCreatedAt}-{sumTotalReleases}-{urlHash}"`       |
| `/api/seasons/anime`                              | `W/"sa-{year}-{SEASON}[-b{bgmId}]-{count}-{lastMs}-{urlHash}"`    |
| `/api/anime/groups`                               | `W/"ag-{bgmId}-{count}-{lastMs}-{urlHash}"`                       |

公式确定性：同一查询在数据未变时 ETag 完全相同，多副本部署也能命中。

## 验证 304

```bash title="curl 验证"
H=$(curl -sI "https://anibt.net/rss/anime.xml?bgmId=543360&groupSlug=kirara-fantasia")
ETAG=$(echo "$H" | grep -i '^etag:' | cut -d' ' -f2- | tr -d '\r')

curl -sI -H "If-None-Match: $ETAG" \
  "https://anibt.net/rss/anime.xml?bgmId=543360&groupSlug=kirara-fantasia" | head -1
# → HTTP/2 304
```

## 时间轴：max-age + s-maxage + SWR + 304 怎么交互

很多人会问："`stale-while-revalidate=86400`（24 小时）是不是意味着我可能拿到 24 小时前的旧数据？"——**不会**。

### 角色

- **U**：你的 RSS 客户端（NetNewsWire / Inoreader / 自建脚本）
- **CDN**：Cloudflare 边缘节点
- **O**：origin（AniBT 服务器）

### 端点示例：`/rss/anime.xml`（`max-age=300, s-maxage=300, swr=86400`）

```txt title="时间轴"
T=0s    U 第一次请求
        CDN 没缓存 → 回源 O，O 拿到数据库快照 → 返回 200 + ETag=A + body
        CDN 把响应缓存 300 秒（fresh 期），再延伸 86400 秒（stale 期）
        U 拿到 200 + body + ETag=A
        U 本地缓存 300 秒

T=60s   U 又来（轮询太勤）
        U 自己的浏览器 / 客户端缓存还在 max-age 内 → 根本不发请求
        (origin 0 命中，CDN 0 命中)

T=400s  U 来（第二次实际请求）
        U 本地 max-age=300 已过期 → 带上 If-None-Match: A 发请求
        CDN s-maxage=300 也已过期，但还在 swr=86400 内
        CDN 决策：立即把缓存里的旧版本（含 ETag=A）返回给 U；
                 同时异步发 1 个请求到 O 刷新
        U 收到响应：服务器看 If-None-Match: A == 当前 ETag → 304 Not Modified
        (用户感知：0 延迟拿到 304，知道数据没变 → 不用解析 XML)

        与此同时 CDN 异步回源：
          - 数据真没变 → O 直接返回 304 → CDN 把 fresh 期再续 300s
          - 数据有更新 → O 返回 200 + ETag=B + 新 body → CDN 替换缓存

T=700s  U 来（第三次）
        U 带 If-None-Match: A
        CDN 已有新缓存（ETag=B 或仍是 A）
        CDN 把 ETag=B 的 body 返回 → 但 U 的 If-None-Match 不匹配 → 200 + 新 body
        U 拿到新数据
```

### `swr=86400` 的真正意义

<Callout type="info">
  **不是**"用户会被强制忍受 24 小时旧数据"——而是"如果 origin 当前不可用 / 回源慢，CDN **最多** 用 24 小时内的旧版本兜底，避免给用户报错"。
</Callout>

只要 origin 健康，**你永远拿到不超过 `s-maxage`（5 分钟）旧的数据**：

- T=400s：CDN 立即吐 5–10 分钟前的版本（你大概率收到 304，无感）
- T=400s 后几百毫秒：CDN 已经异步刷成最新版本
- T=700s 你下一次来：直接拿最新

## `cf-cache-status` 含义

| 状态      | 含义                                          | origin 是否被打 |
| --------- | --------------------------------------------- | --------------- |
| `HIT`     | CDN 内存命中，仍在 fresh 期                   | 否              |
| `STALE`   | CDN 给旧版本（SWR 期内）+ 后台异步刷         | 异步打 1 次     |
| `MISS`    | CDN 没缓存，同步回源                         | 同步打 1 次     |
| `EXPIRED` | 缓存过期且**超过** SWR 窗口，必须同步回源      | 同步打 1 次     |
| `BYPASS`  | 跳过缓存（一般不会发生在公开端点）            | 同步打 1 次     |

## 完整 curl 演示

```bash title="一次完整验证"
# Step 1: 第一次请求，拿 ETag + 看缓存头
curl -sSI 'https://anibt.net/rss/anime.xml?bgmId=543360' \
  | grep -iE '^(HTTP|etag|cache-control|cf-cache-status)'
# HTTP/2 200
# cache-control: public, max-age=300, s-maxage=300, stale-while-revalidate=86400
# etag: W/"r12-1776624843260-9k2x4f"
# cf-cache-status: MISS               ← 第一次穿透到 origin

# Step 2: 立刻再来一次（仍在 max-age 内，但用 -H 强制触发条件请求）
ETAG='W/"r12-1776624843260-9k2x4f"'
curl -sSI -H "If-None-Match: $ETAG" \
  'https://anibt.net/rss/anime.xml?bgmId=543360' \
  | grep -iE '^(HTTP|cf-cache-status)'
# HTTP/2 304                          ← 数据没变
# cf-cache-status: HIT                ← CDN 内存里直接命中

# Step 3: 等 6 分钟后再来（s-maxage 已过期，进入 SWR 期）
sleep 360
curl -sSI -H "If-None-Match: $ETAG" \
  'https://anibt.net/rss/anime.xml?bgmId=543360' \
  | grep -iE '^(HTTP|cf-cache-status|age)'
# HTTP/2 304
# cf-cache-status: STALE              ← CDN 给旧版本，后台已异步刷新
# age: 367                            ← 这份缓存已存在 367 秒
```

## Python 客户端最小实现

```python title="poll.py"
import requests, time

URL = 'https://anibt.net/rss/anime.xml?bgmId=543360&groupSlug=kirara-fantasia,haruhana&language=CHS'
state = {'etag': None, 'last_modified': None}

def poll():
    headers = {}
    if state['etag']:          headers['If-None-Match']     = state['etag']
    if state['last_modified']: headers['If-Modified-Since'] = state['last_modified']

    r = requests.get(URL, headers=headers, timeout=10)
    if r.status_code == 304:
        print(f"[{time.strftime('%H:%M:%S')}] 304 not modified, body 0 bytes")
        return None  # 数据没变，不用解析
    if r.status_code == 200:
        state['etag']          = r.headers.get('ETag')
        state['last_modified'] = r.headers.get('Last-Modified')
        print(f"[{time.strftime('%H:%M:%S')}] 200 OK, body {len(r.content)} bytes, "
              f"cf-cache-status={r.headers.get('cf-cache-status')}")
        return r.content
    if r.status_code == 429:
        retry = int(r.headers.get('Retry-After', '60'))
        print(f"rate-limited, sleeping {retry}s")
        time.sleep(retry); return poll()
    r.raise_for_status()

# 每 15 分钟轮询一次
while True:
    poll()
    time.sleep(15 * 60)
```

这套写法满足：

- 第一次响应缓存 ETag → 之后每次走 304，**带宽几乎为零**
- CDN 给 `STALE` 时你也不知道（也不需要知道），用户体验完全相同
- 触发 429 自动按 `Retry-After` 退避
