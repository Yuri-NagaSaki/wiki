## 各端點的上限一覽

| 端點                    | `limit` 預設 | `limit` 上限 | 後端候選掃描上限 | 速率限制（預設）   |
| ----------------------- | ------------ | ------------ | ---------------- | ------------------ |
| `/rss/anime.xml`        | **100**      | **1000**     | **4000** 條候選  | 120 次 / 60 秒 / IP |
| `/rss/group/<slug>.xml` | 100          | **200**      | —                | 120 次 / 60 秒 / IP |
| `/rss/magnets.xml`      | 100          | **200**      | —                | 120 次 / 60 秒 / IP |
| `/api/subtitle-groups`  | —            | 全量返回     | —                | 同上 60 秒視窗      |
| `/api/seasons/anime`    | —            | —            | —                | 30 次 / 60 秒 / IP  |
| `/api/anime/groups`     | —            | —            | —                | 30 次 / 60 秒 / IP  |

<Callout type="info">
  **`limit` 越界處理**：超過上限會被自動**截斷到上限**，不會報錯。例如 `?limit=5000` 給 `/rss/anime.xml` 實際返回 1000 條。
</Callout>

### `MAX_OVERSCAN`（僅 `/rss/anime.xml`）

當你疊加 Tag 過濾（resolution / language / subtitle / format / episode / episodeKey）時，後端先按索引拉候選，再在記憶體裡篩 Tag。**候選池上限是 4000 條**，避免單次請求掃庫失控。

絕大多數番劇 + 字幕組組合的實際資料量都遠小於這個值；如果你過濾極其嚴格（例如某番劇某組只有 3 集 1080p 簡中外掛），後端拉到 4000 條後就停止，再做 Tag 篩選並返回 `limit` 條。**實操影響**：極冷門組合可能"看起來少"——這是設計上的安全閥。

## 提高上限

<Tabs items={['客戶端層面', '服務端層面（自部署）', '觸發 429']}>
  <Tab value="客戶端層面">
    1. **拉滿 `limit`**：把 URL 的 `?limit=` 改成端點上限（1000 / 200）。一次請求拿更多，等於把 N 次輪詢合成 1 次。
    2. **用 304 替代輪詢**：每次請求帶 `If-None-Match`。服務端確認資料沒變就返回 304 空 body，**不計入很多客戶端的"流量統計"**，但仍會消耗 1 次速率配額。
    3. **拉長間隔**：預設建議 ≥ 5 分鐘。番劇釋出頻率本身按"集數"計算，每分鐘掃一次沒有意義。
    4. **合併併發**：不要對同一 URL 併發發多個請求，快取層會讓後續請求拿到一樣的結果，但白白消耗速率額度。
  </Tab>
  <Tab value="服務端層面（自部署）">
    透過 Convex / 部署平臺的**環境變數**調整（重啟後生效）：

    ```bash title="env"
    # 把每 IP 60 秒 120 次提高到 60 秒 600 次
    RSS_ANIME_RATE_LIMIT_MAX=600
    RSS_GROUP_RATE_LIMIT_MAX=600
    RSS_MAGNETS_RATE_LIMIT_MAX=600

    # 或者改視窗為 10 秒（變成 120 次 / 10 秒）
    RSS_RATE_LIMIT_WINDOW_MS=10000

    # 關閉 CDN 邊緣快取（不推薦，僅測試用）
    RSS_CACHE_TTL_SECONDS=0
    ```

    <Callout type="warn">
      `/rss/anime.xml` 的 `MAX_LIMIT=1000` 和 `MAX_OVERSCAN=4000` 是原始碼常量（`convex/rssFeed.ts`），**不透過環境變數調整**。如果你的私有部署需要更高，直接改原始碼後 `npx convex deploy`。
    </Callout>

    提高之後需要觀察：

    - **單次響應 XML 大小**：1000 條約 0.8–2 MB，注意頻寬。
    - **Convex query 時長**：候選 4000 條 + 記憶體 Tag 過濾通常 < 200ms；繼續上調 `MAX_OVERSCAN` 接近函式 1 秒上限時會有失敗風險。
    - **客戶端解析時長**：很多 RSS reader 預設只保留最近 N 條，請同步調整。
  </Tab>
  <Tab value="觸發 429">
    - 響應包含 `Retry-After: <seconds>`，按這個退避，**不要立即重試**。
    - 短期內被打 429 通常是配置錯（輪詢太快 / 多個 reader 用同 URL）。把間隔放大到 5 分鐘 + 啟用 304，幾乎不會再觸發。
  </Tab>
</Tabs>

## 1000 使用者場景下的真實負載

> **核心結論**：origin 真實承受的回源率與併發使用者數**幾乎無關**，只與 `s-maxage` 和查詢引數多樣性有關。

請求流轉：

```txt
使用者 ──► 瀏覽器 / RSS 客戶端快取 (max-age)
        ──► Cloudflare CDN (s-maxage + SWR)
            ──► origin (Convex / TanStack SSR) ──► 資料庫
```

- CDN 內未過期 → **0 origin 請求**
- CDN 過期但仍在 SWR 視窗 → 使用者拿舊版本（0 延遲），origin 收到 **1 個**非同步重新整理請求
- 完全過期 → origin 收到 1 個同步請求（~50–200ms 回源）
- 客戶端帶 `If-None-Match` → ETag 命中時 origin 僅返回 304（無 body，幾乎零開銷）

各端點在 **1000 使用者、客戶端預設 15 分鐘輪詢** 假設下的預估負載：

| 端點                                     | 客戶端日請求量    | origin 真實回源（次/小時） | 備註                                          |
| ---------------------------------------- | ----------------- | -------------------------- | --------------------------------------------- |
| `/rss/anime.xml`（全站）                  | ~96k              | **~12**                    | 5min CDN 視窗，全部使用者共享同一快取            |
| `/rss/group/<slug>.xml`                  | 取決於 slug 數    | **~12 × slug 數**          | 50 個 slug ≈ 600 次/小時                      |
| `/rss/magnets.xml`（簡單）                | —                 | **~30**                    | 全部使用者用相同篩選時                          |
| `/rss/magnets.xml`（複雜、按 query 分快取） | —                 | **接近 1× 唯一查詢數 / 60s** | 每人查詢條件不同時 CDN 命中率 ≈ 0，SWR 緩解延遲 |
| `/api/subtitle-groups`                   | ~96k              | **~12**                    | 資料極少變化，幾乎全 304                       |
| `/api/seasons/anime`                     | ~48k              | **~6**                     | 季度資料周更，長快取                           |
| `/api/anime/groups`（按 bgmId）           | —                 | **~12 × 熱門番劇數**        | 冷門番劇首次回源較多                          |

**綜合**：1000 使用者全套訂閱 + 前端使用，origin 一小時實際承受 **< 1000 次請求**，其中 90%+ 是 304。SWR 讓"使用者感知延遲"和"origin 回源壓力"完全解耦——回源慢不會拖慢使用者。

<Callout type="warn">
  **絕對不要做的事**：
  - 多執行緒併發拉同一 URL（白白消耗速率配額，CDN 會讓結果完全相同）
  - 關閉 ETag / `If-None-Match`（每次都拿全量響應，浪費頻寬）
  - 間隔 < 1 分鐘（必觸發 429，且對你沒任何收益）

## 主流 RSS 客戶端的實際行為

不同客戶端預設的輪詢頻率、ETag/304 支援差異很大。**好訊息**：本站對所有客戶端都返回正確的 `ETag` + `Last-Modified` + `Cache-Control`，符合 HTTP 標準的客戶端會自動走 304，**你不用做任何配置**。

| 客戶端                              | 預設輪詢           | `If-None-Match` | 尊重 `Cache-Control: max-age` | 備註                                            |
| ----------------------------------- | ------------------ | --------------- | ----------------------------- | ----------------------------------------------- |
| **NetNewsWire**（macOS/iOS）        | 1 小時             | ✅              | ✅                            | 表現最好，幾乎全 304                             |
| **Inoreader**（雲端）               | ~30 分鐘           | ✅              | ✅                            | 自帶服務端聚合，1000 使用者共享同一上游請求         |
| **Feedly**（雲端）                  | ~1 小時            | ✅              | ✅                            | 同上，對 origin 極友好                          |
| **FreshRSS**（自建）                | 使用者配置（預設 1h）| ✅              | ✅                            | 推薦配置 ≥ 15min                                |
| **Tiny Tiny RSS**（自建）           | 使用者配置           | ✅              | ⚠ 部分版本無視                 | 建議手動設 ≥ 15min                              |
| **Reeder**（macOS/iOS）             | 跟隨 Inoreader/Feedly | —            | —                             | 透傳第三方聚合服務                              |
| **qBittorrent RSS 自動下載**         | **預設 30 分鐘**   | ❌（每次拉全量） | ❌                            | **對 origin 最不友好**，但配合 CDN 仍 OK         |
| **Transmission + Flexget**          | 使用者配置           | ✅（Flexget 支援）| ✅                           | 推薦                                            |
| **自建指令碼（Python `requests`）**    | 你寫多少就多少     | 取決於實現      | 取決於實現                    | 見 [Python 客戶端最小實現](/docs/open-api/caching#python-客戶端最小實現) |

### 針對場景的預期

- **個人訂閱 1–2 個番劇 RSS（NetNewsWire / Inoreader）**：每小時 1 個請求，全部 304，origin 幾乎感知不到。
- **qBittorrent 用 RSS 規則自動下載**：每 30 分鐘拉一次全量 XML（約 50–200 KB），但因為 CDN 命中，實際打到 origin 也只是 5 分鐘一次。
- **公開聚合站（Inoreader / Feedly）**：1000 個訂閱者後端共享同一份訂閱，對 origin 等同 1 個使用者。

### 調優建議

- 用 **Inoreader / Feedly / NetNewsWire iCloud** 這類支援雲端聚合的客戶端 → 你和其他使用者共享快取
- qBittorrent / Transmission 直接拉的話，把規則間隔調到 ≥ 15 分鐘（預設 30 分鐘也很合理）
- 自建指令碼必須實現 `If-None-Match` 複用，否則等於浪費頻寬

## 後端釋出 → 使用者看到的完整延遲拆解

以"字幕組釋出一個新種子"到"訂閱者的 RSS 客戶端彈出通知"為例，逐段算一遍最壞情況：

```txt title="延遲拆解"
T=0       字幕組提交 release 到 AniBT 後臺
          (Convex mutation 寫入 releases / animeReleases / files 表)
T=0       資料庫立即可讀（Convex 強一致）
          /rss/anime.xml 此刻請求會拿到新資料，但 CDN 還快取著舊版本

T≤300s    CDN 快取（s-maxage=300）視窗內：
          客戶端請求被 CDN 攔截，仍拿舊版本
          → 這一段是使用者感知延遲的主要來源

T=300s    CDN 快取進入 SWR 期：
          下一個使用者請求觸發 CDN 非同步回源
          原 origin 拿到新資料 → CDN 快取更新

T=300~600s 客戶端的下次輪詢週期（取決於客戶端）：
          - 若客戶端輪詢間隔 = 5min → 最快下一次拿到新版本
          - 若間隔 = 15min → 平均再等 7.5 min
          - 若間隔 = 1h → 平均再等 30 min
```

### 典型場景的端到端延遲（中位數）

| 客戶端配置                  | 後端釋出 → 使用者看到        |
| --------------------------- | -------------------------- |
| 自建指令碼 5 min 輪詢 + ETag  | **3–5 分鐘**               |
| NetNewsWire 預設 1h 輪詢    | 30–35 分鐘（瓶頸在客戶端） |
| Inoreader 雲端聚合          | 15–30 分鐘（瓶頸在 Inoreader 自己的輪詢頻率） |
| qBittorrent 預設 30min      | 15–30 分鐘                 |

### 最佳化哪一段最有效

1. **縮短客戶端輪詢間隔（最有效）**：從 1h 改成 5min，立刻把延遲從 30min 砍到 5min。
2. **縮短 `s-maxage`（次要）**：從 300s 改成 60s，origin 壓力漲 5 倍但使用者延遲只少 4 分鐘，價效比低。
3. **強制 origin 立即推送（不可行）**：HTTP RSS 是 pull 模型，沒辦法。

<Callout type="info">
  如果你需要**實時**（< 1 分鐘）通知，RSS 不是合適方案，應該用 webhook / Telegram bot / 站內推送。RSS 的設計目標就是"**最終一致 + 極低成本**"，5–30 分鐘的延遲是它的功能不是 bug。
</Callout>
